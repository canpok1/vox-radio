# 0054. godog による BDD e2e テストの導入

- ステータス: 採用
- 日付: 2026-06-10

## コンテキスト

全体リファクタリングを予定しており、内部構造を大きく変えても「プロダクトの機能が壊れたこと」を検知できる安全網が必要になった。既存テストはユニットテスト中心で、CLI 境界をまたぐ主要動線（init・check 群・episodegen 各ステップ/一括・feedgen・slackpost・キャッシュ連携）を通しで検証するテストがなかった。またテストケースを Gherkin 記法の仕様書としてまとめたいという要望があり、仕様書とテストが乖離しない仕組みが求められた。

## 決定

Cucumber 公式 Go 実装の godog を導入し、`e2e/features/*.feature`（日本語 Gherkin）をそのままテストとして実行する。テストは `//go:build e2e` タグで通常の `go test ./...` から分離し、`make e2e` と CI（build.yml の並列ジョブ `e2e`）で実行する。CLI は TestMain でビルドしたバイナリを os/exec で起動する方式とし、シナリオごとに独立した作業ディレクトリ・環境変数で実行する。外部依存のうち LLM（OpenAI 互換）・VOICEVOX・RSS フィード・Slack API は httptest のモックサーバーで差し替え、ffmpeg/ffprobe のみ実物を使う（CI で apt インストール、ローカルに無い場合は `@ffmpeg` タグ付きシナリオを自動スキップ）。fake LLM はプロンプト冒頭の見出し行（`# [A] 記事選別プロンプト` 等）でステップを判別し、各ステップの JSON Schema に適合する固定レスポンスを返す。

## 結果

主要動線 26 シナリオが約4秒で完走する安全網が得られ、リファクタリング中の機能破壊を CLI 境界・成果物ファイル・HTTP 境界の観測点で検知できる。Gherkin が仕様書兼テストになるため乖離が構造的に起きない。一方、依存（godog とその間接依存）が増え、ステップ文⇔Go 関数のマッピングという間接層でデバッグの手間がやや増える。fake LLM がプロンプト見出しに依存するため、見出し変更時は `e2e/fake_llm.go` の更新が必要（プロンプト変更の検知網も兼ねる）。

## 検討した代替案

- **Gherkin は文書、テストは素の Go**: 依存ゼロで既存の標準 testing 流儀と一致するが、仕様書とテストの対応を運用で維持する必要があり乖離リスクが残るため却下。
- **`cli.NewRootCmd()` の in-process 実行**: キャッシュ・ログ・init が cwd 相対で `os.Chdir` がプロセスグローバルになりシナリオ独立性を壊すこと、main の配線や exit code を検証できないことから却下。
- **CI で実 VOICEVOX コンテナを起動**: 起動が重く external image への依存が増える。合成結果の音質は e2e の検証対象でないためモックで十分と判断し却下。
