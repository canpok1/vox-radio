# 0049. 組み込みプロンプトの品質を LLM-as-judge で評価するフレームワークを導入する

- ステータス: 採用
- 日付: 2026-06-07

## コンテキスト

組み込みプロンプト（`internal/cli/prompts/*.md`）の内容品質を、変更のたびに人手で確認するのは難しい。既存の品質担保は出力スキーマ検証＋自己修復リトライ（ADR-0002 / ADR-0003）のみで、「プロンプトが期待する仕事をしているか」は測れていない。LLM は Gemini 無料枠を使うため、評価がレートリミットで開発 CI を止めることは避けたい。第一弾は入力が自己完結しテストケースを作りやすい校正プロンプト `proofread.md` を対象とする。

## 決定

プロンプト出力を Gemini に複数観点（各 1〜5 点）で採点させ、平均が閾値以上で合格とする LLM-as-judge 評価フレームワークを `internal/eval` に新設する。実行は build tag `eval` ＋週次 schedule / 手動 dispatch に限定し、モックのみの開発 CI（PR / push）とは分離する。品質不足のときだけ fail とし、API・レートリミット等のエラーは inconclusive（`t.Skip`）扱いとする。最低実行間隔は既存 `llm.Client` の throttler を流用する。テストデータは二層構成とする（固定の回帰セット = `expectation` 付き・全件実行、プールから週番号 seed でサンプリングする汎化セット = judge 自律判定）。

## 結果

内容品質の自動監視が可能になる。回帰セットで既知の致命誤読を守りつつ、汎化セットの seed ローテーションで固定データへの過適応と多様性不足を緩和できる。開発 CI と分離したためレートリミットが開発を止めない。一方で judge の主観・揺れに依存し、汎化セットは judge モデル精度に左右される（強めの judge モデルを指定可能にして緩和）。実 API を使うため定期実行は無料枠・ネットワークに依存するが、inconclusive 扱いで影響は小さい。

## 検討した代替案

- **決定的な正解照合のみ（Gemini 不使用）**: 安定・低コストだが多観点の品質・自然さを測れず要件を満たさないため不採用。
- **固定キュレーションのみ**: 再現性は高いが過適応・多様性不足の懸念が残るため不採用。
- **開発 CI（PR / push）へ統合**: 常時ゲートできるが無料枠レートリミットで開発が止まるため不採用。
- **辞書からの動的生成**: 多様性は最大だが誤読生成ロジックの構築コストが高く第一弾の範囲外として見送り。
