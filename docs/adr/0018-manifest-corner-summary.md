# 0018. コーナー単位サマリーを manifest に追加し履歴キャッシュへ集約する

- ステータス: 採用
- 日付: 2026-06-01

## コンテキスト

`manifest.json` には番組全体サマリー（`summary`）と各コーナーの記事参照（`corners[].articles[].{title,url}`）しかなく、コーナー単位で「どんな会話をしたか」を表す情報が無い。配信メタデータとして会話情報をある程度載せたいという要望に加え、issue139（番組単位の履歴キャッシュ `.vox-radio/cache/{program_id}.jsonl`）が目指す「過去エピソードのその後言及」では、過去に話した内容を参照したい。

しかし既存の要約は、記事ソースを要約した `RundownArticle.Summary/Points` か、番組全体を1つにまとめた `summary` のみで、**コーナー粒度の会話内容**を表すものが存在しない。長尺番組ではコーナー粒度の参照ができず、その後言及の材料として粒度が不足する。

## 決定

- 各コーナーの台本セリフ（`03_lines.json` の `CornerLines.Lines`）から、LLM でコーナー単位の `summary`（要約文）と `points`（要点配列）を生成する。実装は既存の `LLMProgramSummarizer` をコーナー単位に展開したものとする。
- 生成結果は `manifest.json` の `ManifestCorner.Summary` / `ManifestCorner.Points` に格納し、**manifest を会話サマリーの単一ソース**とする。
- 専用のキャッシュ機構は新設しない。issue139 の履歴 JSONL 集約が manifest のコーナーサマリーを取り込む形にし、キャッシュは 139 に集約する（本対応は 139 をブロッカーとする）。

## 結果

- 会話内容ベースの「その後言及」材料が得られ、配信メタデータにコーナー粒度の情報を載せられる。
- manifest を単一ソースとすることで、同じ情報を複数箇所に重複保持せずに済む。既存 summarizer 資産を再利用でき実装コストが小さい。
- トレードオフ: コーナー数だけ LLM コールが増え、コスト・レイテンシ・レート制限の影響が増える。139 完了がブロッカーとなり並行着手できない。

## 検討した代替案

- **既存ラウンダウン記事要約（`RundownArticle.Summary/Points`）を流用**: LLM コール不要だが「ソース記事の要約」であって「会話の要約」ではなく、会話情報掲載・その後言及という目的に合致しないため却下。
- **コーナーサマリー専用キャッシュを新設**: 139 と二重実装になり保守コストが増える。manifest を経由すれば 139 に集約できるため却下。
- **全体サマリーのみ（現状維持）**: コーナー粒度の参照ができず、長尺番組で粒度が不足するため却下。
