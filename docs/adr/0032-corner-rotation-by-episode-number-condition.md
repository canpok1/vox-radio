# 0032. コーナーの回ごと組み替えを回番号条件で制御し、採用結果を rundown に乗せる

- ステータス: 採用
- 日付: 2026-06-04

## コンテキスト

毎回同じコーナー構成だと飽きやすいため、回ごとに出るコーナーを組み替えたい。一方でオープニング・エンディングのような毎回必須のコーナーは固定で出す必要がある。要件は「複数候補の中からその回に出すコーナーを切り替える（例: 4候補のうち2つ）」「固定コーナーは毎回出す」「どの候補を出すかを事前に確定でき、再実行しても変わらないこと」。

既存構造では、コーナーは `EpisodeSpec.Corners []CornerConfig` の順序付き配列で定義され（ADR-0011）、`02_rundown.json`（`model.Rundown`）が回ごとの内容の single source of truth（ADR-0015）、回番号はキャッシュから算出される。直前の ADR-0031 では「ときどき登場するゲスト」を回番号条件で制御する方式を採用した。

## 決定

- コーナー配列内に **選択グループ**（`candidates` を持つエントリ）を導入する。グループは配列の特定位置に置かれ、採用された候補がその位置に展開される。`candidates` を持たない通常コーナーは毎回採用される固定コーナーとする。
- 候補の採否は各候補の `condition`（`episodes` 明示リストと `every` 周期の**論理和**）による**回番号からの決定論的処理**（乱数なし）。確率・乱数方式は採らない。
- 出現条件の型 `EpisodeCondition`（`episodes` / `every` / `Matches`）を **ゲスト（ADR-0031 / #231）と共有**し、重複実装を避ける。将来 `select_count`（候補からN個）や確率方式を `condition` 配下に**後方互換で追加**できる拡張点とする。
- 採用コーナーの解決はコーナーを消費するエントリポイントで行う。full pipeline（`episodegen`）では算出済み回番号で解決して `Spec.Corners` に反映し、collect/rundown/script/manifest を採用コーナーで一貫させる。単独 `rundown` は回番号をキャッシュから算出して解決、単独 `collect` は全候補（superset）を収集、単独 `script` は `rundown` の採用コーナーのタイトルから再構成する（回番号不要で再現安定）。
- 採用結果は **rundown の既存 `Corners` にそのまま乗せる**。ゲストと異なり新しいモデルフィールドは追加しない。採用コーナーで rundown を生成すれば、その回の内容として自然に永続化される。
- 回番号が不明（キャッシュ無効等で 0）の場合は**全候補を採用**し（内容を欠落させない degraded mode）、警告する。
- 番組全体の目標時間による time budget 制御は本決定では扱わない（各コーナー `length_sec` 合計が暗黙の全体時間という現状を維持）。

## 結果

**良い影響**: どの回にどのコーナーが出るか事前に確定でき計画しやすい。採用コーナーで rundown を生成するため `script` 単独再実行でも構成が再現する。新モデルフィールド不要で永続化がシンプル。`EpisodeCondition` 共有でゲストとの一貫性と将来拡張性を両立する。固定コーナーは構造上必ず採用され、選択グループはスロット位置を保つ。

**トレードオフ**: 回番号条件はキャッシュ有効（`cache.enabled` かつ `program.id`）が前提で、単独 `rundown` に回番号算出の追加が必要。候補数の制御（N個に絞る）や確率方式は本決定では提供せず `condition` 拡張に委ねる。collect/rundown まで解決が波及するためゲストより配線箇所が広い。

## 検討した代替案

- **確率（ランダム）方式**: 意外性は出せるが事前確定・再現ができず計画しづらい。確率は将来の `condition` 拡張に委ねて却下。
- **採用結果を rundown の新フィールド／専用中間ファイルへ永続化**: コーナーは元々 rundown の構成要素であり、採用コーナーで rundown を生成すれば既存 `Corners` に自然に乗る。新フィールドは冗長なため却下。
- **選択グループを corners 配列外の別フィールドに置く**: コーナーは出現位置（OP〜各コーナー〜ED）が意味を持つため、順序付き配列内のスロットとして表現する方が自然。配列外は位置情報を失うため却下。
- **個数指定（select_count で候補からN個）方式**: 「4候補から2つ」を直接表現できるが、本機能は計画しやすさを優先し回番号条件に統一。将来の拡張点として残す。
