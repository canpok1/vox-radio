# 0032. コーナーの回ごと組み替えを回番号条件で制御し、採用結果を rundown に乗せる

- ステータス: 採用
- 日付: 2026-06-04

## コンテキスト

毎回同じコーナー構成だと飽きやすいため、回ごとに出るコーナーを組み替えたい。一方でオープニング・エンディングのような毎回必須のコーナーは固定で出す必要がある。要件は「複数候補の中からその回に出すコーナーを切り替える（例: 4候補のうち2つ）」「固定コーナーは毎回出す」「どの回にどのコーナーが出るか事前に確定でき、再実行しても変わらないこと」。

既存構造では、コーナーは `EpisodeSpec.Corners []CornerConfig` の順序付き配列で定義され（ADR-0011）、`02_rundown.json`（`model.Rundown`）が回ごとの内容の single source of truth（ADR-0015）、回番号はキャッシュから算出される。直前の ADR-0031 では「ときどき登場するゲスト」を回番号条件で制御する方式を採用した。

## 決定

- 各コーナー（`CornerConfig`）に任意の `condition` を追加する。`condition` を持たないコーナーは毎回出る固定コーナー、`condition` を持つコーナーは回番号が合致した回だけ出る。専用のグループ（候補をまとめる入れ子構造）は導入せず、順序付き配列内で各コーナーに条件を付けるだけで組み替えを表現する。
- 採否は `condition`（`episodes` 明示リストと `every` 周期の**論理和**）による**回番号からの決定論的処理**（乱数なし）。確率・乱数方式は採らない。
- 出現条件の型 `EpisodeCondition`（`episodes` / `every` / `Matches`）を **ゲスト（ADR-0031 / #231）と共有**し、重複実装を避ける。将来 `select_count`（候補から個数指定）や確率方式は、必要になった時点で `condition` 拡張ないしグループ構造の再導入で対応できる拡張余地とする。
- 採用コーナーの解決はコーナーを消費するエントリポイントで行う。full pipeline（`episodegen`）では算出済み回番号で解決して `Spec.Corners` に反映し、collect/rundown/script/manifest を採用コーナーで一貫させる。単独 `rundown` は回番号をキャッシュから算出して解決、単独 `collect` は全コーナー（superset）を収集、単独 `script` は `rundown` の採用コーナーのタイトルから再構成する（回番号不要で再現安定）。
- 採用結果は **rundown の既存 `Corners` にそのまま乗せる**。ゲストと異なり新しいモデルフィールドは追加しない。採用コーナーで rundown を生成すれば、その回の内容として自然に永続化される。
- 回番号が不明（キャッシュ無効等で 0）の場合は**全コーナーを採用**し（内容を欠落させない degraded mode）、警告する。
- 番組全体の目標時間による time budget 制御は本決定では扱わない（各コーナー `length_sec` 合計が暗黙の全体時間という現状を維持）。

## 結果

**良い影響**: どの回にどのコーナーが出るか事前に確定でき計画しやすい。採用コーナーで rundown を生成するため `script` 単独再実行でも構成が再現する。新モデルフィールド不要で永続化がシンプル。`EpisodeCondition` 共有でゲストとの一貫性と将来拡張性を両立する。フラットな `condition` のみのため `CornerConfig` の追加は1フィールドで済み、グループ専用の入れ子構造・検証ルールが不要。固定コーナー（`condition` なし）は構造上必ず採用され、条件付きコーナーは配列位置を保って出現する。

**トレードオフ**: 回番号条件はキャッシュ有効が前提で、単独 `rundown` に回番号算出の追加が必要。「この候補群から必ずN個」という個数指定（`select_count`）はフラットな条件では表現できず、必要になればグループ構造の再導入が要る。collect/rundown まで解決が波及するためゲストより配線箇所が広い。

> 注: 本 ADR 採用当時はキャッシュ有効化を `cache.enabled` かつ `program.id` 設定で制御していたが、ADR-0043 で `cache.enabled` を廃止し `program.id` を必須化、キャッシュを常時有効とした。現在は `program.id` 必須・キャッシュ常時有効が前提となる。

## 検討した代替案

- **確率（ランダム）方式**: 意外性は出せるが事前確定・再現ができず計画しづらい。確率は将来の `condition` 拡張に委ねて却下。
- **選択グループ（`candidates` で候補を入れ子にまとめる）方式**: 「1スロットから候補を選ぶ」「候補からN個（`select_count`）」を明示的に表現できるが、本機能は条件ベース・個数指定なしで確定しており、各コーナーに `condition` を付けるだけで順序を保ったまま同等の組み替えが表現できる。入れ子構造・グループ専用の検証ルール（グループ entry の制約、ネスト禁止、top-level 条件の扱い）が不要になるためフラットな条件方式を採用。個数指定が必要になれば将来再導入する。
- **採用結果を rundown の新フィールド／専用中間ファイルへ永続化**: コーナーは元々 rundown の構成要素であり、採用コーナーで rundown を生成すれば既存 `Corners` に自然に乗る。新フィールドは冗長なため却下。
