# 0034. 出現条件 EpisodeCondition に剰余（offset）を追加しN者ローテーションを可能にする

- ステータス: 採用
- 日付: 2026-06-04

## コンテキスト

ゲストやコーナーを N 個で順番にローテーションさせたい（例: A/B/C を 1回目→A、2回目→B、3回目→C…）。ゲスト・コーナーが共有する `EpisodeCondition`（ADR-0031 / 0032）の `every: N` は `episodeNumber % every == 0`、すなわち**余り0のグループ（3,6,9…）しか表現できず**、余り1（1,4,7…）・余り2（2,5,8…）を作れない。`episodes: [...]` の手書きは有限のため継続番組で破綻する。

なお ADR-0033 で remainder 方式は一度却下したが、それは「`not`（補集合）の表現手段として」剰余を使う案が補集合と非対称だったため。本件の用途は補集合の代用ではなく「ローテーションの位相指定」であり、却下理由は当てはまらない。

## 決定

- `EpisodeCondition` に `Offset int`（YAML `offset,omitempty`）を追加し、`Matches` の `every` 判定を `episodeNumber % every == offset` に変更する。未指定（0）は倍数回のままで**完全後方互換**。
- `every: 3` + `offset: 0/1/2` の3条件で全回を均等分担しローテーションを表現する。
- 検証で `offset < 0` / `offset > 0 && every == 0` / `offset >= every` をエラーにする。`not` 内の `offset` も既存の再帰検証で扱う。
- 共有型のためゲスト・コーナー双方に同時適用。判定は回番号からの決定論的処理（乱数なし）で、新たな永続化は追加しない。

## 結果

**良い影響**: `every` + `offset` だけで N者ローテーションを宣言的に書ける。`omitempty` の1フィールド追加かつ未指定で従来挙動と一致するため後方互換。`Matches` の式変更のみで `ResolveCornersForEpisode` / `guest.Select` 側は不要。全回を漏れ・重複なく分割でき計画性・再現性も維持する。

**トレードオフ**: N 条件を各エントリに記述する必要があり「グループ構造」より冗長（ADR-0032 と同じ割り切り）。`offset` は `every` 依存のため単独指定や `offset >= every` は検証で弾く。回番号の起点（1始まり）と剰余の対応は設定者が把握する必要がある。

## 検討した代替案

- **ローテーション専用セクション新設**: 宣言的で直感的だが既存 `condition` と二重になり、入れ子構造・専用検証・学習コストが増える。フラットな `condition` を選んだ ADR-0032 と一貫させ却下。
- **0始まり方式（offset:0 を 1,4,7… に割当）**: 直感的だが、未指定時の `every` 単独挙動（倍数回）と判定式がねじれ後方互換の説明が複雑になる。後方互換を優先し余り方式を採用。
