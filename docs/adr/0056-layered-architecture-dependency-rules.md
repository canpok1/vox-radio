# 0056. レイヤードアーキテクチャと依存方向ルールの明文化

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

層の境界が暗黙的なため歪みが生じていた（model への YAML ロード混入、ドメイン内部での設定ロード、pipeline interface が具象型 `*assemble.Result` を返す、台本生成の暗黙のファイル契約など）。ルールの明文化も検証手段もなかった。

## 決定

`docs/development/architecture.md` を新設し明文化する。

1. 層構造（cli → pipeline → ドメイン → config → model → 基盤）と許可 import を表で定義し `go list` で検証する
2. 単一ドメイン専用 spec は各ドメインへ、共有設定のみ config へ置く
3. ステップ間データは戻り値で渡し、暗黙のファイル契約を禁止（中間ファイルは pipeline / cli が書く）
4. interface の引数・戻り値に具象ドメイン型を使わない

現存違反は同書の「既知の違反」節に列挙し、順次解消する。

## 結果

- 依存ルールを機械的に確認でき、レビュー基準が明確になる
- model が純粋化し、構造体注入に統一されてテストのファイル依存が減る
- 違反節の保守と横断依存追加時の表更新という運用コストが生じる

## 検討した代替案

- **spec ロードを config へ集約**: Slack 固有知識が config に漏れ肥大し、model 純粋化も未達のため却下
- **暗黙ファイル契約の文書化のみ**: fake がファイルを書かないとテストが落ちる脆弱性が残り却下
- **.claude/rules/ のみに配置**: 人間が見落としやすいため本体は docs/development/ に置く
