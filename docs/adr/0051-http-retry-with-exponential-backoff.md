# 0051. 外部 HTTP API 呼び出しに指数バックオフのリトライを導入する

- ステータス: 採用
- 日付: 2026-06-08

## コンテキスト

LLM API（Gemini/OpenAI 互換）・VOICEVOX・記事収集（HTML/RSS）への HTTP 呼び出しは、サーバーが一時的に 503（overloaded）などを返すと即座にエラーとして扱い、パイプラインが停止していた。特に LLM API は過負荷時に 503 を返すことが多く、少し待って再試行すれば回復するケースが大半であるため、一過性の障害でパイプライン全体が落ちるのは耐障害性として不十分だった。

各クライアントの HTTP 呼び出し箇所（`internal/script/llm/client.go`・`internal/synth/client.go`・`internal/collect`）は構造が異なるが、いずれも `*http.Client` を利用している点は共通していた。

なお LLM クライアントの既存 `MaxRetries` は「JSON スキーマ検証失敗時の再生成リトライ」であり、HTTP レイヤのリトライとは別概念である。

## 決定

共通パッケージ `internal/httpretry` に、リトライ機能を持つ `http.RoundTripper`（`Transport`）を新設し、各クライアントの `http.Client.Transport` に挿す方式を採用する。

- **対象ステータス**: 5xx 全般 + 429（レート制限）。それ以外（2xx/4xx の 429 以外）はリトライしない。
- **バックオフ**: 指数バックオフ（初期 1 秒 → 2 秒 → 4 秒、上限 8 秒）、最大 3 回リトライ。パラメータはコード内定数で固定（設定 YAML 化はしない）。
- **context 連動**: バックオフ待機中に context がキャンセルされたら即中断する。
- **ボディ再送**: POST 等のボディ付きリクエストは `req.GetBody` でボディを再構築して再送する。
- **適用範囲**: 生の `*http.Client` を使う LLM（OpenAI 互換）・VOICEVOX・記事収集（HTML/RSS）。`collect.New(nil)` は従来 `http.DefaultClient`（共有）を使っていたが、共有クライアントを汚さないためリトライ Transport 付きの専用クライアントを生成する。

### 適用しない範囲

Dify Chat（`internal/script/llm/dify_chat.go`）と Slack（`internal/slack/poster.go`）はサードパーティ SDK 経由で HTTP ステータスを直接扱えないため、本 RoundTripper の対象外とする。Dify はユーザー視点では LLM API だが、SDK 側のリトライ機構／エラー判定が別途必要になるため本決定のスコープ外とし、必要なら別途対応する。

## 結果

一過性の 5xx/429 でパイプラインが停止しにくくなり、耐障害性が向上する。リトライ判定とバックオフのロジックが 1 箇所（`internal/httpretry`）に集約され、各クライアントは Transport を挿すだけで透過的にリトライが効くため、呼び出し側コードやステータス判定ロジックの変更が不要になった。

トレードオフとして、恒久的な 5xx の場合はリトライ分だけ失敗確定までの時間が延びる（最大 1+2+4=7 秒程度）。また既存テストで 5xx を返してエラーを検証していたものはリトライ待ちで遅くなるため、リトライ対象外の 4xx を返す形へ変更した（5xx/429 のリトライ挙動は `httpretry` の単体テストで網羅）。

## 検討した代替案

- **各呼び出し箇所に `Do(client, req)` ヘルパー関数を挟む方式**: 明示的だが全呼び出し箇所の書き換えが必要で、ステータス判定ロジックとの二重管理になりやすいため却下。RoundTripper 方式なら呼び出し側を変更せず透過的に適用できる。
- **リトライ回数・待機時間を設定 YAML 化**: 現時点で運用上の調整需要がなく、固定値で十分。設定面を増やす複雑さを避けるため見送り（必要になれば後から拡張可能）。
- **ネットワークエラー（接続失敗等）もリトライ対象に含める**: 今回のスコープは 5xx/429 に限定。`http.Transport` 自体が接続レベルの再試行を一部行うため、まずは応答ステータスベースのリトライに絞った。
