# 0005. 配信を Podcast(RSS) に一本化する

- ステータス: 採用
- 日付: 2026-05-30

## コンテキスト

毎日 1 回・約 5 分の番組を配信する。視聴側の入口（Podcast アプリ、Slack 等）ごとに専用の配信実装を持つと、API 連携や認証の保守が増える。一方で過去回を無制限に残すとストレージが肥大する。

## 決定

配信は **Podcast(RSS フィード) に一本化**する。`feed.xml` と音声ファイルを HTTP で公開するだけの構成とし、視聴側はその feed URL を購読する。**Slack は標準の RSS アプリで購読**（`/feed subscribe <feed-url>`）し、Slack 専用 API は実装しない。過去回は **直近 7 件 / 7 日**のみ保持し、feed には 7 件だけ列挙、7 日より古い音声は削除する。

## 結果

- 配信経路が 1 つになり、公開先に置くだけで Podcast アプリ・Slack の双方から購読できる。専用実装・認証が不要。
- 保持を 7 件に絞ることでストレージが一定に保たれる。ダウンロード済みの回は購読者端末に残るため聴き逃しは起きにくい。
- 限定公開が必要な場合は URL に推測困難なトークンを含めるプライベート Podcast 方式で対応できる。
- リアルタイム性の高いプッシュ通知のような体験は持たない（RSS のポーリング前提）。

## 検討した代替案

- **Slack 専用 Bot 配信**: チャンネル体験は作り込めるが、Slack API の保守が必要で配信経路が分散するため却下。
- **複数配信チャネルの併用**: 到達範囲は広がるが実装・運用コストが増え、RSS で大半を賄えるため却下。
