# 0019. 番組構成外の会話情報を会話メモとして manifest に残し履歴へ集約する

- ステータス: 採用
- 日付: 2026-06-01

## コンテキスト

番組生成では、rundown（記事の事実: `url`/`title`/`summary`/`points`）に含まれない会話情報が多数生まれる。キャラの近況・掛け合い、記事への感想、番組中のハプニング・アドリブ、継続ネタなど多岐にわたるが、台本に生成されるだけでどこにも保存されず破棄されていた。

既存の要約はいずれも「事実に基づいた記事内容」に限定され、番組要約 `LLMProgramSummarizer` は話者情報を捨てて本文のみを LLM へ渡していた。結果 `prompts/write.md` の `{{past_episodes}}` には記事ベース要約しか流れず、次回番組で会話の連続性（一貫性・コールバック・その後言及）を担保できない。後から会話を思い出せる形で幅広く残したい。

## 決定

- 番組要約ステップ（`LLMProgramSummarizer`）を拡張し、**話者付き台本**から番組要約と「会話メモ」を **1 LLM コールで同時抽出**する。専用ステップは新設しない。
- 会話メモは特定カテゴリに固定せず、**rundown に含まれない会話情報全般**を対象とし、自由カテゴリ付きの開放的なメモ配列で持つ。

  ```go
  type ConversationNote struct {
      Category     string   `json:"category"`      // 自由記述（近況/掛け合い/感想/ハプニング/継続ネタ ...）
      CharacterIDs []string `json:"character_ids"` // 関係するキャラID（speaker_role準拠）
      Note         string   `json:"note"`
  }
  ```

- 会話メモは `manifest.json` の `conversation_notes` に格納して **manifest を単一ソース**とし（ADR-0018 踏襲）、cache が取り込んで `write.md` の過去エピソード文脈として LLM へ注入する。
- 蓄積は既存の JSONL append＋直近 N 件方式を踏襲し、設定も既存 `cache` を流用する。専用トグルやキャラ最新状態のマージ保持は導入しない。

## 結果

- 記事に紐づかない会話情報が回をまたいで参照可能になり、キャラの一貫性・コールバック・その後言及が効く。配信メタにも会話メモが載る。
- 既存 summarizer を再利用するため LLM コール数は増えず、実装コストが小さい。manifest 単一ソースにより重複保持を避けられる。
- トレードオフ: 番組要約プロンプトの責務が「要約＋会話メモ抽出」に増え出力が複雑化する。メモの粒度・品質は LLM 依存で、長尺番組では件数増でトークンを圧迫しうる（直近 N 件制限で緩和）。

## 検討した代替案

- **固定カテゴリ（近況/掛け合い/継続ネタ）に限定**: 感想・ハプニング等を取りこぼし「会話を思い出す」目的に届かないため却下。自由カテゴリの開放配列にした。
- **会話メモ抽出用の専用 LLM ステップを新設**: 番組あたりのコールが増えコスト・レート制限で不利。既存要約に統合できるため却下。
- **cache 専用フィールドに保持し manifest 非公開**: ADR-0018 の manifest 単一ソース方針と二重管理になり却下。会話メモも配信メタとして公開する。
- **台本セリフ全文を保存**: トークン肥大。要約済みメモで足りるため却下。
- **キャラ最新状態をマージ保持**: マージ判断が複雑で既存 append-log と非対称。直近 N 件で足りるため却下。
