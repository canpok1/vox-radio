# 0013. フィードパーサを自作XMLパーサからgofeedへ移行する

- ステータス: 採用
- 日付: 2026-05-31

## コンテキスト

`internal/collect/rss.go` はフィード取得に自作の `encoding/xml` デコーダを使用しており、RSS 2.0（`<rss><channel><item>`）のみに対応していた。Atom 形式（`<feed><entry>`）のフィードを指定すると、パースエラーにならず0件が返るサイレント障害が発生していた（Issue #81）。

フィード取得の対応形式を拡大するにあたり、自作パーサの拡張か、実績あるライブラリへの置き換えかを検討した。

## 決定

`github.com/mmcdole/gofeed v1.3.0` を採用し、自作パーサを置き換える。バージョンは参照実装リポジトリ（canpok1/ai-feed）と同一バージョンに固定した。

既存のHTTPクライアントDI（`fp.Client = c.client`）とcontextキャンセル（`ParseURLWithContext`）はgofeedのAPIで維持できるため、httptest ベースのテストとの互換性が保たれる。

## 結果

- **良い影響**: RSS 2.0・Atom・RSS 1.0・JSON Feed を自動判別して統一的にパースできる。フォーマット追加時のメンテナンスコストがなくなる。
- **悪い影響**: 依存パッケージが増加する（gofeed 本体＋ goxpp・goquery 等の間接依存）。
- **トレードオフ**: 自作パーサは依存レスだが、Atom 対応を追加するには XML namespace 処理など相応の実装コストがかかる。

## 検討した代替案

- **自作パーサを Atom 対応に拡張する**: `encoding/xml` で Atom スキーマ構造体を追加する案。RSS・Atom を個別にデコードして統合する処理が必要で、実装・テストのコストが高い。参照実装で実績のあるgofeedと比べメリットが薄いため却下。
- **`gorss` 等の別ライブラリ**: 更新頻度・スター数・機能網羅性でgofeedが勝るため却下。
