# 0081. 起動時に .env ファイルから環境変数を読み込む

- ステータス: 採用
- 日付: 2026-06-16

## コンテキスト

vox-radio の利用には環境変数の設定が必須である。LLM の API キーは設定ファイルの `api_key_env`（`internal/cli/util.go`）、Slack の Bot トークンは `bot_token_env`（ADR-0080）で名前を間接指定して参照する。利用者はこれらを `export` でシェルに設定する必要があり、シェルやセッションごとの環境変数設定に不慣れな利用者には導入のつまずきになりやすい。

devcontainer では既に `.env`（`.devcontainer/.env-template`）で秘密情報を流し込む運用があり、`.env` の概念はプロジェクト内で馴染みがある。CLI ツールでは起動時に `.env` を読むのが定番で、導入ハードルを下げる効果が見込める。さらに CI では本番／検証など環境ごとに env ファイルを差し替えたいケースがあり、読込対象のパスを指定できる手段も必要になる。なお `os.Getenv` の呼び出しは CLI 層に限る規約（`docs/development/architecture.md`）があり、これを崩さずに実現する必要がある。

## 決定

`github.com/joho/godotenv` を依存に追加し、CLI 層 root の `PersistentPreRunE` で env ファイルを読み込む。読込対象は永続フラグ `--env-file`（デフォルト `.env`）で指定する。

- フラグ未指定時はカレントディレクトリの `.env` を対象とし、存在すれば読み込み、無ければ無視する。
- フラグを明示指定した場合はそのファイルのみを読み込み（`.env` は読まない＝差し替え）、ファイルが存在しなければエラーとする（タイプ・パス誤りを fail-fast で検知）。
- いずれの場合も、env ファイルが存在して解析に失敗した場合はエラーとする。
- 既存の OS 環境変数を優先する（`godotenv.Load` のデフォルト挙動。`Overload` は使わない）。`export` 済みや CI で注入された値を env ファイルが上書きしない。
- 読込を root に置くことで全サブコマンドに一律で効き、読込後は既存の `os.Getenv` 経路がそのまま値を取得するためドメイン層の改修は不要。
- env ファイルは秘密情報を含むため git 管理から除外する（ルート `.gitignore` で `.env` を除外済み。別名ファイルを使う場合は利用者が除外する）。

## 結果

- 利用者は `.env` にキーを書くだけで起動でき、`export` が不要になり導入ハードルが下がる。
- `--env-file .env.prod` のように環境ごとの env ファイルを差し替えられ、CI での本番／検証切替に対応できる。
- 読込点が root に集約され、「`os.Getenv` は CLI 層のみ」の規約を維持できる。
- OS 環境変数優先のため CI/シェル設定が env ファイルに勝ち、意図せぬ上書き事故が起きにくい。
- 依存が 1 つ増えるが、軽量で広く使われるライブラリのため保守コストは小さい。
- 明示指定と未指定でファイル不在時の挙動（エラー／無視）が異なるが、誤指定の検知とゼロコンフィグ起動を両立するための意図的な差である。

## 検討した代替案

- **自前の簡易パーサで実装**: 依存を増やさずに済むが、コメント・クォート・エスケープ等のエッジケースを自前で保守する必要があり、定番ライブラリに対する優位がない。却下。
- **`.env` を優先（`godotenv.Overload`）**: env ファイルが常に OS 環境変数を上書きし、CI/シェル設定が効かなくなる上書き事故のリスクがあるため却下。
- **`.env` 固定でフラグを設けない**: 実装は最小だが、環境ごとの env ファイル差し替えができず CI 運用の要件を満たせないため却下。
