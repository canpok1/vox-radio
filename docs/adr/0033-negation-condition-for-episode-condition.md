# 0033. 出現条件 EpisodeCondition に否定（補集合）条件 not を追加する

- ステータス: 採用
- 日付: 2026-06-04

## コンテキスト

コーナー（ADR-0032）・ゲスト（ADR-0031）の出現条件 `EpisodeCondition` は、`episodes`（明示リスト）と `every`（N の倍数回）の**論理和（OR）**で「合致したら採用」する肯定条件のみをサポートする。

これだけでは「あるコーナーを行わない回に別のコーナーを行う」という**補集合の組み合わせ**が表現できない。具体例として「5回に1回行うコーナーA（`every: 5`）」と「コーナーAを行わないときに行うコーナーB（=5の倍数ではない回）」を実現したいが、後者の「5の倍数ではない」を書く手段がない。既存の `episodes`/`every` はどちらも肯定（合致＝採用）で、否定（合致しない回で採用）を表す条件が欠けている。

## 決定

- `EpisodeCondition` に任意の否定条件 `Not *EpisodeCondition`（YAML キー `not`）を追加する。`not` に指定した条件に**合致する回は採用しない**（補集合）。`EpisodeCondition` 自身を再帰的にネストできるため、任意の条件の補集合を表現できる。
- `Matches` のセマンティクスを次のとおりとする。
  1. `not` が設定され、その条件に合致する回は**除外**（`false`）する。
  2. 肯定条件（`episodes` / `every`）の論理和に合致すれば採用する。
  3. 肯定条件が**両方未指定**の場合は「常に真」とみなす（`not` 単独指定を許し、純粋な補集合「○○ではない回すべて」を表現可能にするため）。
- 検証（`validateEpisodeCondition`）は、`episodes` / `every` / `not` の**いずれか一つ以上**が設定されていることを要求する（従来は episodes か every の必須だった条件を緩和）。`not` が設定されていれば中身を**再帰的に検証**する。
- `EpisodeCondition` はコーナーとゲストで共有しているため、`not` は**両機能に同時に適用**される。判定は引き続き回番号からの**決定論的処理**（乱数なし）。
- 否定方式の表現として、条件全体を反転する `negate: true` フラグ案、および「コーナーAが出ない回」を参照する `unless_corner` 参照案ではなく、**ネスト可能な `not` 条件**を採用する。
- 回番号が不明（0）のときの degraded mode（コーナーは全採用・ゲストは非出演）の挙動は ADR-0031 / ADR-0032 のまま変更しない。

## 結果

**良い影響**: 「5の倍数ではない回」のような補集合を `not: { every: 5 }` で素直に書ける。肯定条件と否定条件を組み合わせられる（例: `every: 2` ＋ `not: { episodes: [6] }` =「2回ごと、ただし6回目は除く」）。`EpisodeCondition` への 1 フィールド追加かつ `omitempty` のため**完全に後方互換**で、既存の `episodes`/`every` 設定はそのまま動く。コーナー・ゲスト両方に同時に効く。`Matches` は再帰呼び出しで実装でき、`ResolveCornersForEpisode` / `guest.Select` 側の変更は不要。

**トレードオフ**: 肯定条件を両方省くと「常に真」に変わるため、`Matches` のセマンティクスが「肯定条件なし＝判定不能」から「肯定条件なし＝常に真（not で絞る）」へ変化する（検証で空条件は引き続き弾くため実害は小さい）。`not` のネストを深くした条件は可読性が下がりうるが、典型用途は 1 段で足りる。`not: { every: 1 }`（=決して採用されない）のような論理的に無意味な組み合わせはバリデーションでは弾かない（書き手の責任）。

## 検討した代替案

- **`negate: true` フラグ方式**: `condition` 全体の真偽を反転するだけのため最もシンプルだが、「肯定条件＋否定条件の組み合わせ」（例:「2回ごと、ただし6回目は除く」）が表現できない。1 軸の反転に留まり拡張性が低いため却下。
- **コーナー参照方式（`unless_corner: "コーナーA"`）**: 「Aが出ない回」を自動追従でき A/B の同期ズレを防げるが、コーナー間に依存・順序・循環参照チェックという新たな複雑さが生じ、プロジェクトの「各コーナー独立・決定論的」方針（ADR-0032）から外れる。条件を独立に書く方針では `not` でA の条件をそのまま複製すれば足りるため却下。
- **`every` に余り（remainder）を持たせる方式**: `every: 5, remainder: 0` と `remainder: [1,2,3,4]` のように剰余で表現する案。「5の倍数でない」を単一値で表せず（補集合をリストで列挙する必要があり）、補集合の概念と対称でない。`episodes` の補集合も別途必要になり一貫しないため却下。
