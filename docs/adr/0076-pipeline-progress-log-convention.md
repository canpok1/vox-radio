# 0076. パイプライン進捗ログの出力規約を定める

- ステータス: 採用
- 日付: 2026-06-16

## コンテキスト

`episodegen` パイプラインは collect / rundown / script / synth / summary / assemble の各ステップが `slog` で進捗ログを出すが、規約が無く出力が不揃いだった。`script/write`・`script/direct` は「開始」だけ出して対応する「完了」が無く、`rundown` は開始も完了も出さない（collect→script 間が無言になる）。メッセージは大半が日本語なのに `excluded past articles` / `cache entry appended` / `pipeline complete` だけ英語で、完了時の経過秒も collect/synth/assemble は `完了 (…, T.Ts)` 形式、summary 系は `elapsed_s=<生float>` 属性とフォーマットが分かれていた。今後ステップを追加するたびに同じ不揃いが再発する恐れがあり、横断的な規約が必要になった。

## 決定

パイプライン各ステップのログ出力規約を以下に定める。

- **開始/完了はペアで出す**: ログを出すステップ・サブステップは「開始」と「完了」を必ず対で出力する（開始のみ・完了のみを禁止）。
- **完了の経過時間はメッセージに埋め込む**: `完了 (…, T.Ts)`（`%.1fs`）形式に統一する。経過秒を独立した `elapsed_s` 属性で出さない。
- **メッセージ本文は日本語**: 人間向けのメッセージ本文は日本語に統一する。
- **属性キーは英語**: `key=value` の属性キー（`count` `program_id` `corner` `err` 等）は英語のまま維持し、grep・機械処理性を保つ。
- **進捗は stderr / 最終結果は stdout**: 進捗ログは `logger`（stderr＋ファイル、タイムスタンプ・step 付き）に出し、パイプラインの最終結果行（`pipeline complete`）のみ `fmt.Printf` で標準出力(stdout)に出す。両者を混在させない。

## 結果

- **良い面**: 各ステップのログが揃い、開始だけ・完了だけの非対称や日英混在が解消する。新規ステップ追加時も従う指針ができ、再発を防げる。stderr=進捗 / stdout=結果の分離が明文化され、スクリプトから最終結果だけを安定して読める。
- **移行/悪い面**: 既存の英語メッセージ（`excluded past articles` / `cache entry appended` / `pipeline complete`）を日本語化するため、ログ文言をアサートするテスト（`internal/rundown/rundown_test.go`）と e2e フィーチャ（`e2e/features/episodegen_full.feature`）の検証文字列更新が必要になる。
- **トレードオフ**: メッセージ本文は日本語、属性キーは英語という二言語混在を許容する。本文の可読性（日本語話者向け）とログのパース性（英語キー）を両立させるための割り切り。

## 検討した代替案

- **親ステップのみ開始/完了を出し、サブステップの開始ログ（script/write・script/direct）は削除する**: 出力は簡潔になるが write/direct の進行が見えなくなり、長時間ステップの体感が悪化するため却下した。
- **属性キーも含めて全面日本語化する**: 見た目は最も一貫するが、ログの grep・機械処理がしづらくなるため却下した。
- **最終結果行も logger 経由（stderr）に寄せる**: 全行をタイムスタンプ付きで揃えられるが、「最終結果を stdout に出す」設計が失われ、結果だけを読むスクリプト連携・e2e の標準出力検証が成り立たなくなるため却下した。
