# 0004. 中間生成物をファイルで疎結合に繋ぎ、入出力規約を定める

- ステータス: 採用
- 日付: 2026-05-30

## コンテキスト

パイプラインは collect → script(多段) → synth → assemble → publish と多段で進む（ADR 0003）。各ステップをメモリ上で一体に繋ぐと、不調なステップだけの再実行ができず、モジュールを独立して実装・テストしにくい。また 1 日に複数回試行する運用があり、生成物を取り違えず容易に掃除したい。

## 決定

各ステップの出力をファイルに残し、**ステップ単位で再実行可能**にする。入出力規約は次のとおり:

- 個別サブコマンドは**入出力パスを明示指定**する（`--in`/`--out`、synth は `--out-dir`、assemble は `--clips` 等）。
- 指定ディレクトリが存在しなければ**自動生成**する。
- 一括 `run` は**出力ディレクトリのみ指定**（`--out-dir`、既定 `output/`）。**最終成果物 `episode.mp3` は直下**、**中間生成物は `<out-dir>/intermediate/` 配下**にまとめ、ファイル名は生成順の連番プレフィックス付きで正準固定（01_articles.json / 02_summaries.json / 03_lines.json / 04_script.json / 05_clips/clip_NNN.wav）。
- **日付固定ディレクトリは作らない**（複数回実行に対応。ディレクトリ名は呼び出し側が決める）。掃除はディレクトリ単位で `rm -rf`。

Podcast の guid/pubDate（`episode-<date>`）は publish 側で日付指定（既定=実行日）から決め、作業ディレクトリ構成からは独立させる。

## 結果

- 中間生成物が残るためデバッグと局所再実行が容易。各モジュールを固定 fixture に対して独立実装でき、Issue の依存も最小化できる。
- 中間物が `intermediate/` に隔離され、最終物と区別して一括削除できる。
- 日付に縛られないため 1 日複数回の試行が衝突しない。
- 一時ファイルが増えるため `output/` は `.gitignore` 対象にする。

## 検討した代替案

- **日付付きランディレクトリ規約**: 掃除単位は明確だが 1 日複数回実行で衝突し、命名の自由度も下がるため却下。
- **メモリ上で受け渡す一体型**: 高速だが局所再実行・独立実装ができず、デバッグも困難なため却下。
