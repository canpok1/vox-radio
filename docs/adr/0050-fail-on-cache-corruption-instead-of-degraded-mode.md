# 0050. キャッシュ破損時は degraded mode ではなくエラー停止する

- ステータス: 採用
- 日付: 2026-06-07

## コンテキスト

ADR-0031/0032/0033 では、回番号が不明（0）のとき degraded mode（コーナー全採用・条件付きキャスト/ゲスト非出演）で続行し警告する方針だった。

しかし回番号0が発生する経路は、`rundown` の `loadCacheEntries`（`internal/cli/util.go`）がキャッシュ `Load()` のエラーを握りつぶして0を返す場合のみ。ファイル不存在は `Load()` がエラーにならず回番号1（初回扱い）になるため、回番号0は実質「キャッシュ破損／読込障害」だけを意味していた。

この挙動には2つの問題があった。利用者は破損とファイル不存在を区別できず誤解しやすい。また破損を無視して続行すると回番号の連続性が壊れ、過去に生成した番組（mp3）との繋がりが失われる。加えて `episodegen` は既に `Load()` エラーで停止しており、`rundown` だけが degraded mode に入る不統一があった。

## 決定

`loadCacheEntries` は読込エラーを握りつぶさず呼び出し元へ伝播し、`rundown` もエラー停止する（`episodegen` と統一）。

回番号0が到達不能になるため、degraded mode 専用の警告コード（`util.go` の `selectCasts` / `resolveCorners` 内の `episodeNumber == 0` 警告ブロック）を削除する。`cast.Select` / `ResolveCornersForEpisode` / `EpisodeCondition.Matches` の `episodeNumber <= 0` 防御ガードは安全弁として残す。本決定は ADR-0031/0032/0033 の degraded mode 部分の方針を上書きする。

## 結果

破損を即座に検知でき、修正してから再実行する運用を促せる。回番号の連続性が保たれ、生成済み番組との繋がりが守られる。両コマンドのキャッシュ障害時挙動が統一され、デッドコードも減る。トレードオフとして破損時は番組生成が完走しなくなるが、不整合な番組を作るより停止して原因に気付ける方が望ましい。

## 検討した代替案

- **degraded mode の維持**: 破損とファイル不存在の混同・回番号不連続の問題が残るため却下。
- **破損時に回番号を推定して続行**: 破損時のエントリ数推定は信頼できず、不正確な回番号で番組を生成するリスクがあるため却下。
- **`rundown` だけ警告を強化して続行**: 握りつぶしの根本原因が残り、見落とせば同じ問題が起きるため却下。
