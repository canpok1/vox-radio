# 0002. LLM プロバイダ抽象と OpenAI 互換ワイヤープロトコル

- ステータス: 採用
- 日付: 2026-05-30

## コンテキスト

台本生成（② script）は軽量 LLM を多段で呼び出す。プロバイダ（Gemini / OpenAI / OpenRouter / ローカル等）を将来差し替え可能にしたいが、各社専用 SDK を抱えると分岐が増え保守が重くなる。一方ドメイン層（pipeline）は生成手段の詳細を知るべきではない。各社の構造化出力（`response_format`）対応度にも差がある。

## 決定

ドメイン層は `ScriptGenerator` インターフェースのみを知り、内部のワイヤープロトコルは **OpenAI 互換 1 実装に統一**する。プロバイダ切替は列挙型（`LLM_PROVIDER`）の分岐ではなく **`base_url` ＋ `model` の設定差し替え**で行う。既定は Gemini の最新軽量モデル `gemini-3.1-flash-lite`（2026-05-08 GA）。構造化出力は `json_schema` / `json_object` を使いつつ、各社差を吸収するため **スキーマ検証＋自己修復リトライを必ず併用**する。

## 結果

- 1 つの HTTP クライアント実装で多数のプロバイダに対応でき、分岐が増えない。
- プロバイダ追加は設定変更だけで済む。
- スキーマ検証＋リトライにより、対応度の低いプロバイダでも出力品質を担保できる。
- Gemini の OpenAI 互換レイヤは beta 扱いでネイティブ専用機能（検索グラウンディング等）は使えないが、記事は ① collect で自前収集するため不要。必要時は `Client` 実装を差し替えれば `ScriptGenerator` 境界で吸収できる。

## 検討した代替案

- **列挙型でプロバイダごとに分岐**: 各社の差を明示できるが分岐とテストが増え、追加コストが高いため却下。
- **Gemini ネイティブ API を直接利用**: ネイティブ機能を使えるが他プロバイダへ移れず、互換 API で要件を満たせるため現時点では却下。
