# 0053. slackpost に状態ファイルによる冪等な再投稿を導入する（ADR-0035 の二重投稿方針を改訂）

- ステータス: 採用
- 日付: 2026-06-08
- 補足: 「二重投稿ガードは持たず呼び出し側責務とする」とした [ADR-0035](0035-slack-episode-posting-subcommand.md) の判断を改訂する。slackpost の投稿構成（音声を親メッセージに置く 2 段構成）は維持する。

## コンテキスト

slackpost は親メッセージ（mp3 + 初期コメント）をアップロードし、その `ts` を `files.info` の `shares` から取得してスレッド返信（要約・コーナー）を投稿する。`files.completeUploadExternal` のレスポンスは投稿メッセージの `ts` を含まないため、`ts` 取得は最大 30 秒のポーリング（`pollForTS`）に依存している。Slack には「アップロード状況や投稿 `ts` を同期的に返す API」が存在せず、`files.info` ポーリングまたは `conversations.history` が公式に案内された唯一の回避策である（[python-slack-sdk #1363](https://github.com/slackapi/python-slack-sdk/issues/1363)）。

このため、**アップロードは成功したがポーリングが 30 秒でタイムアウトする**と、スレッド返信が永久に欠落する。ADR-0035 は二重投稿ガードを持たない設計のため、単純に再実行すると**音声が二重投稿**され、安全にやり直せない。毎日 1 回の定常運用で、この「やり直せない欠落」は無視できないギャップである。

## 決定

- **状態ファイルによる冪等な再投稿を導入する**。slackpost は実行段階（①アップロード、②`ts` 取得、③スレッド返信）の進捗を JSON 状態ファイルに記録し、再実行時に未完了段階から再開する。これにより、タイムアウト後の再実行で音声を再アップロードせず、欠落したスレッド返信のみを投稿する。
- **状態ファイルの既定の置き場所は manifest と同じディレクトリ**（`<manifest stem>.slackpost-state.json`）とし、ファイルベース中間成果物の方針（[ADR-0004](0004-file-based-intermediate-artifacts.md)）と一貫させる。slackpost は実行に manifest + mp3 が同ディレクトリへ存在することが必須であり、意味のある再実行ではこのディレクトリが再利用される前提が成り立つ。
- **`--state <path>` フラグで状態ファイルの置き場所を上書き可能にする**。クリーンチェックアウトで実行ディレクトリが毎回破棄される CI でも、状態ファイルを永続パスへ逃がすことで冪等性を保てる。CI 形態が未確定でも、運用側がフラグで吸収できる。
- **状態ファイルには実行の同定キー（mp3 ファイル名・回数）を含め、現在の manifest と一致する場合のみ再開に使う**。異なるエピソードの古い状態ファイルが残っていても誤って再利用せず、新規投稿として扱う。
- **追加の Slack スコープは要求しない**。冪等判定はローカル状態ファイルで完結させ、Slack 側問い合わせ（`files.list` / `conversations.history` 等）による検知は採らない。ADR-0035 の最小スコープ（`chat:write` / `files:write` と、`files.info` が要する `files:read`）を維持する。

## 結果

- タイムアウトや返信失敗が起きても、同じ入力で再実行すれば二重投稿なく投稿が完了する。「安全にやり直せる」状態になる。
- slackpost に状態ファイルという副成果物が増える。実行ディレクトリが再利用される運用では透過的だが、毎回破棄される CI では `--state` で永続先を明示する運用が必要になる。
- ADR-0035 の「二重投稿は呼び出し側責務」は、状態ファイルが参照可能な範囲では本 ADR が肩代わりする。状態ファイルが参照できない再実行（別ディレクトリ・状態破棄）では従来どおり呼び出し側責務に戻る。

## 検討した代替案

- **ポーリングのタイムアウト延長・設定化のみ**: 実装は軽いが、タイムアウトすれば返信欠落・再実行で二重投稿という本質的なギャップが残るため却下。
- **親をテキストメッセージ（`chat.postMessage`）にして `ts` を同期取得**: `ts` レースを根絶できるが、「音声を最初に・最も目立つ位置に出す」という ADR-0035 のコア決定（再生の手間削減）を覆すため却下。
- **Slack 側問い合わせで既投稿を検知**: 実行ディレクトリが破棄される CI にも耐えるが、追加スコープ（`files:read` 相当の一覧・履歴）とマッチングロジックが必要で複雑度が上がるため、まずは状態ファイル + `--state` で対応し将来の拡張余地とする。
