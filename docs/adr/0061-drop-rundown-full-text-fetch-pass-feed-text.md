# 0061. rundown の全文取得を廃止しフィード由来テキストを description として write へ渡す

- ステータス: 採用
- 日付: 2026-06-12

## コンテキスト

ADR-0059 で記事の原文を write（セリフ生成）LLM まで引き渡すことにした。その実装では、rundown 構築時に選別済み記事ごとに記事ページを再取得（`FetchFullText`→`fetchArticle`）し、`<article>/<main>/<body>` から抽出した全文で `Body` を上書きしていた。この全文取得には3つの問題があった。

- **トークンの浪費**: 記事全文は冗長で、write/summarize/flow のプロンプトに丸ごと載るためコストが大きい。
- **ヒューリスティック抽出のノイズ**: `<article>/<main>/<body>` の素朴な抽出はナビ・フッター・関連記事などの雑多なテキストを取り込みやすい。
- **サニタイズ境界の迂回**: collect の `sanitizeArticle`（ADR-0057/0060）が行う不可視文字除去・本文長の切り詰め・インジェクション検出は collect 段階で適用される。rundown の全文取得結果は再サニタイズされないため、長さ上限を超え、注入防御の境界を迂回していた。

加えて、フィードから取得したテキスト（`item.Content`/`item.Description`）と、記事へ直接アクセスして抽出した本文を、いずれも `Body` という同一名で保持しており、由来の区別がつかなかった。

## 決定

rundown の全文取得（`FetchFullText` による `Body` 差し替え）を廃止し、collect が取得したフィード由来テキストをそのまま rundown→write へ流す。あわせて由来で名前を分ける命名規約を導入する。

- **フィードから取得したテキスト = `description`**、**記事へ直接アクセス（個別URL `cfg.Articles` の `fetchArticle`）して取得した本文 = `body`**。`model.Article` / `model.RundownArticle` は `description` / `body`（ともに omitempty）の2フィールドを持ち、生成元に応じて片方のみ設定する。消費側は実効テキスト（`body` 優先・なければ `description`）を返すヘルパー経由で扱う。
- サニタイズ（不可視文字除去・文字数上限・インジェクション検出）は `description` / `body` の両フィールドに適用する。
- `ArticleFetcher` / `FetchFullText` と rundown の fetcher 配線は削除する。`fetchArticle` 自体は個別URL収集で引き続き使う。

本決定は **ADR-0059 の「原文を write に引き渡す」を改訂**する。write へ渡すのは「直接取得した全文」ではなく「フィード由来の description（個別URL の場合は body）」となる。

## 結果

- write/summarize/flow に渡るテキストが短くなりトークンコストが下がる。フィードの description は編集済みでノイズが少ない。
- 注入防御の境界が collect に一本化され、全文取得による迂回（ADR-0057/0060 の防御漏れ）が解消する。
- フィード由来テキストと直接取得本文が名前で区別され、`01_articles.json` / `02_rundown.json` の意味が明確になる。
- トレードオフ: お便り（リスナー投稿）の原文読み上げ（ADR-0059）は、お便りソースのフィードが全文を配信している前提に依存する。フィードが抜粋しか返さない場合は逐語読み上げが崩れうる。
- 破壊的変更: 記事の JSON フィールドが `body` 単一から `description` / `body` の2フィールドへ変わる。

## 検討した代替案

- **全文取得を維持しサニタイズを再適用**: トークン浪費とヒューリスティックのノイズが残り、お便りでも全文の素朴抽出が逐語読み上げに不利。要件（原文の一部を綺麗に渡す）はフィード由来テキストで満たせるため却下。
- **コーナー種別でお便りコーナーのみ全文取得を維持**: 分岐が増え設定・実装が複雑化。全コーナー一律でフィード由来テキストを使う方針を選択。
- **`body` 単一名のまま全文取得だけ廃止**: 由来の区別がつかず、フィードの抜粋を `body`（＝全文の意）と呼ぶ混乱が残るため却下。
