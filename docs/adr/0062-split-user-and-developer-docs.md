# 0062. ドキュメントを利用者向け(README)と開発者向け(docs/development)に分離する

- ステータス: 採用
- 日付: 2026-06-13

## コンテキスト

README が利用者向けと開発者向けの情報を混在させ、最重要の利用手順が巨大な「CLIの使い方」節に埋もれていた。新規ドキュメントの置き場所も暗黙で、手書きのコマンド表と自動生成の `docs/cli/` が重複していた。利用者・開発者・将来の執筆者がそれぞれ迷わない構造と、その方針の明文化が必要だった。

## 決定

- README は利用者向けに限定する（クイックスタート→インストール→使い方→設定方法→各リファレンス→開発リンク）。最短で番組を作れるクイックスタートを先頭に置く。
- 開発者向け（環境構築・ビルド・テスト・プロンプト評価・アーキテクチャルール）は `docs/development/` に集約する。`docs/architecture/` はここへリネームする。
- コマンドドキュメントは cobra の `Short`/`Long` を正として `docs/cli/` に自動生成で一本化し、README の手書き表を廃止する。
- 設定リファレンスは `install --skills` で配布する都合上 `internal/cli/skills/vox-radio/references/` に埋め込んだまま正とし、README/docs はそこへリンクする。
- `make` 系コマンドは開発者専用とし、利用者向けの記述からは除外する。

## 結果

- 利用者の動線が一直線になり、README が約3割スリム化した。
- 「どこに何を書くか」が `docs/README.md` と本 ADR に明文化され、新規執筆者が迷わない。
- コマンド説明の重複が解消し、`Short`/`Long` 更新→`make docs` の単一フローになった。
- 一方で開発者情報が README から分かれ、1ファイル参照では完結しなくなる。

## 検討した代替案

- **開発者向けも README に集約（末尾セクション化）**: README が肥大化し利用者の動線を阻害するため却下。
- **コマンド表を `docs/commands.md` に手書き維持**: `docs/cli/` と二重保守になり乖離リスクがあるため却下。
- **設定リファレンスを `docs/` 配下へ移設**: `go:embed` の配布物と二重持ちになりパス変更も要するため却下。
