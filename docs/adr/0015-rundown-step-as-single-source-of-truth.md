# 0015. 番組構成(rundown)ステップを新設し、扱う記事の正とする

- ステータス: 採用
- 日付: 2026-05-31

## コンテキスト

ADR 0003 は台本生成パイプラインに「[A] テーマ決定(rundown)」を構想したが、実装では `model.Rundown` が定義のみで未使用のまま見送られた。現状の script ステップは collect が集めた**全候補記事を要約**し、ライターが `target_chars` 制約の中で暗黙的に一部だけを採用する。

このため「番組で実際に扱う記事」がどこにも明示的に残らない。結果、ADR 0012 で導入したコンテンツ manifest のコーナー記事一覧が `01_articles.json`（全収集記事）から生成され、番組で扱っていない記事まで出力されてしまう。manifest は外部の配信サービスが RSS を生成する入力となるため、扱った記事のみに絞る必要がある。

## 決定

collect と script の間に **rundown ステップ**を独立サブコマンドとして新設する（既存の「1ステップ=1サブコマンド」方針に沿う）。rundown はコーナーごとに LLM で **(1) 扱う記事の選別 (2) 話の流れ(flow)の設計 (3) 選別記事のみの要約** を行い、`02_rundown.json` を生成する（従来の `02_summaries.json` を置換）。

`rundown.json` を**唯一の正(single source of truth)**とし、script(write→direct) と manifest の双方がこれを参照する。manifest のコーナー記事一覧は選別済み記事のみから生成する。

## 結果

- manifest が番組で扱った記事のみを出力する。
- 記事選別がライター内の暗黙処理から rundown の明示的成果物になり、flow に「どういう想定で選んだか」が残って追跡可能になる。
- 要約対象が選別記事のみになり LLM コストが減る。
- 破壊的変更が生じる: `Scripter.Generate` の引数 `model.Articles`→`model.Rundown`、manifest の `--articles`→`--rundown`、中間ファイル名変更、write プロンプトへの flow 追加。CLI ドキュメント・テストの更新を伴う（ADR 0004 の中間生成物規約に従う）。

## 検討した代替案

- **ライターが採用記事URLを出力**: 局所的だが選別が執筆と密結合になり、設計図(flow)という独立成果物が残らないため却下。
- **台本テキストと記事タイトルの事後マッチング**: 言い換えで一致せず脆いため却下。
- **collect/構成段階で上位N件に固定**: 関連性に基づかず番組内容を粗く変えるため却下。
- **全記事要約は維持し選別ステップのみ追加**: 不採用記事の要約コストが無駄になるため却下。
