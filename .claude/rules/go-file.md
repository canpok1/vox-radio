# Goファイル編集ルール

## 実装方針

`.go` ファイルに変更を加える実装は、テスト駆動開発（TDD）で行うこと。

## 設計フェーズのチェック（実装着手前）

TDD の設計フェーズで以下を確認し、`/simplify` の指摘を待たず設計段階で対処すること。

- **類似処理の重複確認** — 追加する処理と類似の既存処理がないか `grep` でプロジェクト全体を確認する。重複があれば先に共通ヘルパーへ抽出してから実装を始める。
- **共通ヘルパーの早期抽出** — 複数箇所に同種の分岐（例: `switch` による実装選択）が生じる構造が予見される場合は、最初から共通ヘルパーとして `util.go` 等へ設計する。
- **supplementary interface の検討** — 実装ごとに後処理が異なるパターン（例: Push 後の追加操作）が予見される場合は、`Pusher` のような supplementary interface を設計段階で検討する。
- **受け入れ条件のファイル・関数マッピング** — タスクの受け入れ条件を実装着手前にチェックリスト化し、各条件がどのファイル・関数で実現されるかを対応づけてから着手する（条件の見落としを防ぐ）。
- **nil ガード除去前の struct リテラル確認** — struct にフィールドを追加し、パッケージ内で nil ガードを除去する場合は、事前に同パッケージ内の全 `&TypeName{}` リテラル（特に `_test.go`）を `grep -n "&TypeName{"` で洗い出し、デフォルト値を設定してから除去する（nil 参照によるパニックを防ぐ）。
- **中心的なリネーム前の全参照調査** — 型名・フィールド名・定数名・ファイル名をリネーム/変更する場合は、着手前に `grep -rn` で全参照を洗い出し影響範囲を確定する。検索対象は `.go` だけでなく `.md`（ADR・README・`docs/cli/` 等のドキュメント）にも広げ、ドキュメントへの波及も同時に把握する（修正漏れ・self-review/CI での手戻りを防ぐ）。

## 実装時の注意点

- **nilスライスのJSON出力は `null` になる** — JSON で空配列 `[]` を期待する箇所では `var s []T` ではなく `make([]T, 0)` で初期化すること。
- **errcheck 等の主要 linter パターンを実装前に意識する** — `defer resp.Body.Close()` の戻り値無視など、golangci-lint で頻出する指摘パターンを把握しておき、実装後の手戻りを減らすこと。

## 編集後のチェック

`.go` ファイルを編集した後は、以下のチェックを実行して指摘があれば修正すること。修正内容もコミットに含める。

- `gofmt -l .` → 出力があれば `gofmt -w .` で修正
- `golangci-lint run` → 指摘があれば修正
- 依存を追加・変更した場合は `go mod tidy` を実行し、direct/indirect アノテーションを正確に保つこと
