# /simplify 提案の採否ルール

`/simplify`（および `/code-review` 等の類似外部エージェント提案）を採用する際は、以下の基準で判断すること。

## 複数エージェント結果の重複集約（dedup）

`/simplify` は複数エージェントを並列実行するため、同じファイル・同じ箇所への指摘が重複して返ることがある。採否判断の前に以下の手順で集約する。

1. **ファイル名・行番号（または対象シンボル）で指摘を dedup する** — 複数エージェントが同一箇所を指摘した場合、1件に集約する。
2. **集約後の代表1件に対して採否基準を適用する** — 「採用前のチェック」の基準に沿って判断する。

これにより、重複指摘の突き合わせ・二重対応にかかる手間を削減できる。

## 採用前のチェック

- **削除系提案はプロジェクト全体の利用状況を確認する** — 依存・設定・コードの削除提案は、採用前に `grep -r` 等でプロジェクト全体の参照状況を確認する。エージェントは diff のスコープ内で判断するため、他サービス・他スクリプトからの依存を見落とす場合がある。参照がゼロであることを確認してから採用する。
- **動作変更を伴う提案は先に確認してからスキップ判断する** — 戻り値・エラー処理・副作用が変わる提案（例: `json.Valid` 削除でエラーハンドリングが変わるなど）は原則スキップし、スキップ理由を記録する。
- **スコープ外の提案は「方針が明確で小さく直せるか」で扱いを分ける** — 当該 PR・タスクのスコープを超える変更提案でも、方針が明確で小さく直せるものなら即対応する（タスク化しない）。そうでないものはスキップし、その旨を記録したうえで後続タスクとして作成する（詳細は「スコープ外提案の追跡」を参照）。

## スキップ時の記録

採否を判断したら、その理由を作業メモまたはコードレビューコメントに残すこと。

- スキップ例: 「動作変更（`json.Valid` 削除でエラーハンドリングが変わる）のためスキップ」
- スキップ例: 「スコープ外（`*float64 → float64` の型変更は別タスク対応）のためスキップ」

## スコープ外提案の追跡

スコープ外・altitude 指摘をスキップした場合は、作業メモへの記録だけでなく後続タスクとして作成すること（または既存の技術的負債タスクに追記すること）。タスクの作成方法・登録先は `.claude/rules/task-management.md` に従う。

### タスク作成の対象

- 他コマンド・他ファイルに同種の重複が残るもの（例: `readJSON` / `loadProfile` の横展開）
- 配置先・責務分離などの altitude 指摘（「このルールは別ファイルに置くべき」など）

### タスク作成時の記載事項

タスク化するときは「どのメモ / PR のどの指摘か」を背景に明記して、後から再調査コストがかからないようにする。

```
## 背景
worktree-issue-XX.md の /simplify 指摘（スコープ外としてスキップ）より。
{指摘内容の要約}
```

### PR 完了時の棚卸し

PR 完了時（fix-pr 等）に、スコープ外スキップ項目を確認し、未作成のものがあればタスクを作成すること。
