## このスキルの起動タイミング

- **手動実行**: ユーザーが `discuss` を明示的に呼んだとき。
- **LLM 判断による自動起動**: 「〜について相談したい」「〜の仕様を考えてIssueにまとめて」「〜できるようにしたい」など、要望・課題の**相談**を検出したとき。
  - 実装そのものの依頼（「実装して」）では起動しない（相談ではなく実装タスク）。

## 手順

### ステップ1: 相談内容の理解

- 相談・要望の意図と、解きたい課題・期待する成果を把握する。
- 曖昧な場合は `AskUserQuestion` で目的・制約を補う。

### ステップ2: コードベース調査

- 関係する構造体・プロンプト・設定・配線・既存 Issue / ADR を調べる。
- 広域探索は `Agent`（`Explore` / `general-purpose` サブエージェント）を**並列**で起動して効率化する（結論のみ回収）。
- 既存の関連 Issue（`mcp__github__list_issues` / `mcp__github__search_issues` / `mcp__github__issue_read`）・ADR（`docs/adr/`）を確認し、踏襲すべき方針があれば引用する。

### ステップ3: 論点・判断点の整理とユーザー確認（不明点ゼロまで詰める）

**Issue 担当者が判断不要で作業を進められるようにすることが本ステップの目的**。Issue 化（ステップ5）の前に、**残った不明点・未決事項をすべて洗い出して解消し、方針を確定させる**。曖昧さを Issue に持ち越さない。

- 不明点・未決事項を**漏れなく列挙**する（仕様の解釈余地、対象範囲の境界、データ形式、エラー時挙動、命名・配置、依存追加の要否、後方互換性、テスト方針など）。
- 各不明点を次のいずれかで**決着させる**。
  - 「製品としての判断」が必要な点（公開/非公開、対象範囲、コスト増を伴う方式選択など）は **`AskUserQuestion`** で確認する（推奨案を第1選択肢に「(推奨)」付きで提示）。
  - 技術詳細（命名・nil初期化方針・配置など）は既存規約・パターンに沿って**スキル側がデフォルトを決める**（担当者の判断を残さない）。
- **不明点が残っている間は Issue を作成しない**。すべて決着したことを確認してからステップ4へ進む（決着した内容は Issue の「決定事項」へ転記する）。

### ステップ4: アウトプット先の判断（相談の結論）

整理結果から、次のいずれかを選ぶ。

- **新規 Issue 作成** … 実装が必要で既存チケットに該当が無い場合（→ ステップ5へ）。
- **既存 Issue 修正** … 既存チケットのスコープ拡張・決定事項追記で足りる場合（`mcp__github__issue_write` の `method: update`）（→ ステップ5へ）。
- **チケット作成不要** … 相談だけで方針が定まる／見送りの場合。結論と根拠をユーザーに要約して終了する（必要なら `base-tools:work-memo` に残す）。

### ステップ5: Issue 本文の作成 / 修正（新規・既存いずれの場合も）

- **前提（ゲート）**: ステップ3で不明点がすべて解消され、方針が確定していること。未決事項が残っている場合はステップ3に戻り、起票しない。
- 作成は `mcp__github__issue_write`（`method: create`）、既存修正は `method: update` を用いる。
- 構成: `背景・目的` / `現状分析（コードで確認済み）` / `関連（既存Issue・ADR）` / `決定事項（判断不要）` / `仕様（データモデル・配線表など）` / `受け入れ条件（チェックリスト）` / `作業ステップ`。
- 受け入れ条件は変更が必要なファイル・関数にマッピングし、シグネチャ変更の波及を `grep -rn` で確定する（`.claude/rules/go-file.md`）。
- **末尾にフッターを付与**する（CLAUDE.md）。

  ```
  ---
  🤖 Generated with [Claude Code](https://claude.ai/claude-code)
  ```

- ラベルは内容に応じて付与する（例: `enhancement`）。

### ステップ6: ADR 化（必要に応じて）

- アーキテクチャ・技術選定・データ形式など重要判断が含まれる場合は **`create-adr` スキルを起動**してADRを作成し、Issueから参照する（判断したセッションで作成）。
- 些末な選択は対象外（ADRスパム防止）。

### ステップ7: 仕上げ確認（CLAUDE.md「作成後に不備確認」）

- 作成・修正した Issue を読み返し（`mcp__github__issue_read`）、レイアウト崩れ・フッター・受け入れ条件の抜けを確認する。
- ADRを作成した場合は `docs/adr/README.md` のインデックス追記・散文文字数（create-adr の上限）を確認する。

## 設計上の注意

- **実装はしない**: CLAUDE.md「明示的な実装指示がない限りファイル編集・作成をしない」に従い、本スキルは相談・Issue（と必要なら ADR）の作成/修正までを行い、対象機能のコード実装には着手しない。
- **判断はユーザー、詳細はスキル**: 製品判断は `AskUserQuestion` でユーザーに委ね、再現可能な技術詳細はスキルが既存規約から決めて Issue に固定する。
- **不明点ゼロで起票する**: Issue 担当者が判断不要で作業を進められるよう、起票前に不明点・未決事項をすべて洗い出して解消し、方針を確定させてから Issue を作成する（曖昧さを Issue に持ち越さない）。
- **チケット化は結論次第**: 相談の結果として作成不要・既存修正もありうる前提で進める（常に新規Issueを作らない）。
