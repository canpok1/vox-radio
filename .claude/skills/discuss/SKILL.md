## このスキルの起動タイミング

- **手動実行**: ユーザーが `discuss` を明示的に呼んだとき。
- **LLM 判断による自動起動**: 「〜について相談したい」「〜の仕様を考えてタスクにまとめて」「〜できるようにしたい」など、要望・課題の**相談**を検出したとき。
  - 実装そのものの依頼（「実装して」）では起動しない（相談ではなく実装タスク）。

## タスクの作成・更新について

- タスクの登録先・使用ツールは `.claude/rules/task-management.md` に従う（このスキルは「タスクを作成／更新する」という抽象で記述し、具体は当該ルールに集約する）。

## 手順

### ステップ1: 相談内容の理解

- 相談・要望の意図と、解きたい課題・期待する成果を把握する。
- 曖昧な場合は `AskUserQuestion` で目的・制約を補う。

### ステップ2: コードベース調査

- 関係する構造体・プロンプト・設定・配線・既存タスク / ADR を調べる。
- 広域探索は `Agent`（`Explore` / `general-purpose` サブエージェント）を**並列**で起動して効率化する（結論のみ回収）。
- 既存の関連タスク（`mcp__todoist__find-tasks`）・ADR（`docs/adr/`）を確認し、踏襲すべき方針があれば引用する。

### ステップ3: 論点・判断点の整理とユーザー確認（不明点ゼロまで詰める）

**タスク担当者が判断不要で作業を進められるようにすることが本ステップの目的**。タスク作成（ステップ5）の前に、**残った不明点・未決事項をすべて洗い出して解消し、方針を確定させる**。曖昧さをタスクに持ち越さない。

- 不明点・未決事項を**漏れなく列挙**する（仕様の解釈余地、対象範囲の境界、データ形式、エラー時挙動、命名・配置、依存追加の要否、後方互換性、テスト方針など）。
- 各不明点を次のいずれかで**決着させる**。
  - 「製品としての判断」が必要な点（公開/非公開、対象範囲、コスト増を伴う方式選択など）は **`AskUserQuestion`** で確認する（推奨案を第1選択肢に「(推奨)」付きで提示）。
  - 技術詳細（命名・nil初期化方針・配置など）は既存規約・パターンに沿って**スキル側がデフォルトを決める**（担当者の判断を残さない）。
- **不明点が残っている間はタスクを作成しない**。すべて決着したことを確認してからステップ4へ進む（決着した内容はタスクの「決定事項」へ転記する）。

### ステップ4: アウトプット先の判断（相談の結論）

整理結果から、次のいずれかを選ぶ。

- **新規タスク作成** … 実装が必要で既存タスクに該当が無い場合（→ ステップ5へ）。
- **既存タスク修正** … 既存タスクのスコープ拡張・決定事項追記で足りる場合（`mcp__todoist__update-tasks`）（→ ステップ5へ）。
- **タスク作成不要** … 相談だけで方針が定まる／見送りの場合。結論と根拠をユーザーに要約して終了する（必要なら `base-tools:work-memo` に残す）。

### ステップ5: タスク本文の作成 / 修正（新規・既存いずれの場合も）

- **前提（ゲート）**: ステップ3で不明点がすべて解消され、方針が確定していること。未決事項が残っている場合はステップ3に戻り、作成しない。
- 作成・修正は `.claude/rules/task-management.md` のツール・登録先に従う（新規は `mcp__todoist__add-tasks`、既存修正は `mcp__todoist__update-tasks`）。
- `content`（タスク名）は何をするかを簡潔・具体的に書く。
- `description`（タスク本文、Markdown）は次の構成で書く: `背景・目的` / `現状分析（コードで確認済み）` / `関連（既存タスク・ADR）` / `決定事項（判断不要）` / `仕様（データモデル・配線表など）` / `受け入れ条件（チェックリスト）` / `作業ステップ`。
- 受け入れ条件は変更が必要なファイル・関数にマッピングし、シグネチャ変更の波及を `grep -rn` で確定する（`.claude/rules/go-file.md`）。

### ステップ6: ADR 化（必要に応じて）

- アーキテクチャ・技術選定・データ形式など重要判断が含まれる場合は **`create-adr` スキルを起動**してADRを作成し、タスクから参照する。
- 些末な選択は対象外（ADRスパム防止）。

#### ADR は「判断が確定したこのセッション（=タスク作成時）」に作成する（実装へ先送りしない）

- 仕様検討で重要判断を確定させた場合、ADR は **実装フェーズではなく、タスクを作成するこの相談セッションで作成**する（判断の鮮度が高いうちに「なぜそうしたか」を残すため）。
- したがって、タスクの受け入れ条件に「**実装時に ADR 化する**」と書いて先送りしない。ADR は作成と同じセッションで作成し、下記の手順でコミット・PR まで完了させる。
- 例外: 作成時点で設計が固まらず判断を保留した場合のみ、ADR を保留しタスクにその旨（保留理由）を明記する。

#### ADR を作成したら必ずコミットして PR を作成する

ADR ファイル（`docs/adr/NNNN-*.md`）と `docs/adr/README.md` のインデックス追記を作成したら、**そのタイミングで必ずコミットし PR を作成する**こと（ADR を未コミットのまま放置しない）。

1. **作業ブランチを確保する** — `git rev-parse --abbrev-ref HEAD` で現在のブランチを確認し、`main` の場合は**必ず最初に作業ブランチを作成**してから作業する（`main` へ直接コミットしない）。
   ```bash
   git rev-parse --abbrev-ref HEAD   # main なら↓
   git checkout -b <作業ブランチ名>
   ```
2. **コミットする** — ADR ファイルとインデックス追記をコミットに含める。
3. **PR を作成する** — `create-pr` スキル等で PR を作成する（タイトル・本文・フッター付与）。
4. **同一セッションで複数の ADR／関連ファイルを作成した場合は同じ PR にまとめる**（ADR ごとに PR を乱立させない）。

### ステップ7: 仕上げ確認

- 作成・修正したタスクを読み返し（`mcp__todoist__find-tasks`）、本文の構成崩れ・受け入れ条件の抜けを確認する。
- ADRを作成した場合は `docs/adr/README.md` のインデックス追記・散文文字数（create-adr の上限）を確認する。

## 設計上の注意

- **実装はしない**: CLAUDE.md「明示的な実装指示がない限りファイル編集・作成をしない」に従い、本スキルは相談・タスク（と必要なら ADR）の作成/修正までを行い、対象機能のコード実装には着手しない。
- **判断はユーザー、詳細はスキル**: 製品判断は `AskUserQuestion` でユーザーに委ね、再現可能な技術詳細はスキルが既存規約から決めてタスクに固定する。
- **不明点ゼロで作成する**: タスク担当者が判断不要で作業を進められるよう、作成前に不明点・未決事項をすべて洗い出して解消し、方針を確定させてからタスクを作成する（曖昧さをタスクに持ち越さない）。
- **タスク化は結論次第**: 相談の結果として作成不要・既存修正もありうる前提で進める（常に新規タスクを作らない）。
